# Database schema — jae-noi-pork-shop (Cloudflare D1)

Generated by cross-checking `db/schema.ts` against the real production D1 database
(`jae-noi-pork-shop`, binding `DB`) with `wrangler d1 execute --remote`. Kept in
sync as of this writing; re-verify against production if either drifts.

## How migrations actually work here (read this first)

There are **two** SQL sources in this repo and only one of them is real:

- **`migrations/*.sql`** — hand-written, this is the source of truth for the
  actual table/index definitions in production.
- **`drizzle/*.sql`** — output of `drizzle-kit generate` from `db/schema.ts`.
  **Not applied anywhere.** No script runs `wrangler d1 migrations apply`
  against this folder. Treat it as a generated-types/diffing side artifact only.

Migrations are applied and tracked via Cloudflare's native mechanism: run
`npx wrangler d1 migrations apply jae-noi-pork-shop --remote --config
dist/server/wrangler.json` (needs a build first so that config file exists) —
this checks the remote D1's own `d1_migrations` table and applies whatever's
missing. `npx wrangler d1 migrations list jae-noi-pork-shop --remote --config
dist/server/wrangler.json` shows pending ones without applying anything.

**History note (fixed 2026-07-19):** production originally had the full schema
present but its `d1_migrations` tracking table didn't exist at all — the schema
had been applied some other way (most likely `wrangler d1 execute
--remote --file=...` or pasted directly into the D1 console) that bypassed
`migrations apply`'s bookkeeping, so both `0001_orders.sql` and
`0002_storefront_cms.sql` showed as permanently "pending" even though every
table/index already existed. This was low-risk only because every statement in
both files is `IF NOT EXISTS`-safe (tables and indexes alike), so replaying them
was a harmless no-op. Fixed by running `wrangler d1 migrations apply
jae-noi-pork-shop --remote --config dist/server/wrangler.json` once — verified
before/after that `products`/`orders`/`storefront_settings` row counts were
unchanged (4 / 6 / 16), and `d1_migrations` now correctly lists both files as
applied. Tracking works correctly for any future migration from this point on.

To change the schema: write a new `migrations/000N_*.sql` file (matching the
existing style, keep every statement `IF NOT EXISTS`-safe — SQLite has no
`IF NOT EXISTS` form for `ALTER TABLE ADD COLUMN`, so guard those some other
way if you ever need one), then update `db/schema.ts` to match by hand so the
two stay in sync, then run `wrangler d1 migrations apply` against local and
remote. Don't expect `drizzle-kit generate` output to be deployed automatically.

`db/order-repository.ts`, `db/cms-repository.ts`, and `db/storefront-repository.ts`
all query D1 with raw `db.prepare(...).bind(...)` — none of them use the Drizzle
query builder. `db/schema.ts` exists purely for type-safety/documentation and
`drizzle-kit` schema diffing, not as the runtime query layer.

## D1-specific notes for this project

- Foreign keys are always enforced by D1 (can't be turned off) — the
  `order_items.order_id → orders.id` reference is real, not advisory.
- No native boolean/datetime columns are used here — all timestamps are ISO
  strings in TEXT columns (`created_at`, `updated_at`, etc.), not
  `integer(mode:'timestamp')`. Keep new columns consistent with that if you add any.
- `version` columns are the app's own optimistic-concurrency mechanism
  (see `settingsFingerprint`/fingerprint checks in `db/cms-repository.ts`) —
  not a Cloudflare/D1 built-in.
